# 缓存架构

> 本文档描述 Chain Parse Service 中 pool/token 元数据缓存的整体架构：分层模型、数据流向和关键设计决策。

---

## 1. 背景

链上 Swap 日志只携带池子地址和原始金额，不携带 token 元数据（decimals/symbol/name），也不携带池子的 token0/token1。而下游数据（`swaps` 双记账表）需要这些信息才能成立。获取这些元数据的手段只有三种：扫建池事件、发起 eth_call、从 The Graph 预取——三者成本差异巨大，缓存体系的职责就是：**热路径（区块解析）上查元数据必须近乎零成本，真实的 RPC 调用全部隔离到热路径之外。**

---

## 2. 分层模型

```
L1  进程内存        每进程独享,最快,重启即失
L2  Redis          跨进程共享,加速层,可丢弃重建
L3  关系型数据库    唯一权威数据源 (dex_pools / dex_tokens)
```

三条原则：

1. **L3 是唯一的 source of truth**。L1/L2 任何时候都可以从 L3 完整重建。
2. **上层写失败不影响下层**。Redis 不可用只损失加速效果，绝不阻断数据落库。
3. **pool 缓存与 token 缓存分开存储**，与 L3 的两张表一一对应。不合并的原因：token 与 pool 是多对多关系（一个 WETH 出现在数万个池子里），合并意味着元数据大量冗余复制且无法单点纠错。

---

## 3. 数据流

### 写入侧（谁填充缓存）

| 来源 | 时机 | 路径 |
|------|------|------|
| **thegraph-sync**（独立进程） | 周期性批量预取 | subgraph → L3 → L2（批量） |
| **parser 启动预热** | 进程启动 | L3 → L1（同步，阻塞启动）；L3 → L2（异步，不阻塞） |
| **建池事件** | 扫描到 PairCreated/PoolCreated | 事件日志 → L1/L2/L3；token 元数据同步 eth_call 解析 |
| **pending queue worker** | 异步补全 | eth_call → L1/L2/L3 |

启动预热的同步/异步分界：L1 是热路径的第一道防线，必须在引擎处理区块前就位；L2 预热只服务于其他进程和未来重启，不值得阻塞启动。多进程共享同一 Redis 时通过预热标记去重，避免每个进程重复全量预热。

### 读取侧（热路径查找）

| 查询 | 层级 | miss 行为 |
|------|------|----------|
| token 元数据（`fetchTokenMeta`） | L1 → L2 → eth_call | 兜底默认值，保证可用性 |
| pool 身份（`resolvePoolTokens`） | L1 → L2，**到此为止** | 进 pending queue，绝不同步 eth_call |

L2 命中一律回填 L1，同一进程对同一对象只查一次 Redis。

**热路径铁律：swap 解析路径上不允许出现同步 eth_call。** swap 是系统中数量级最大的事件流，任何无界 RPC 都会阻塞整个区块批次。允许同步 eth_call 的位置只有两处，且都有天然的低频保证：建池事件（受限于链上建池速率）和后台 worker（不在扫描线程上）。

---

## 4. Pending Queue：pool 未知时的异步补全

`swaps` 表以 `token_address` 为 InfluxDB tag。tag 是 point 的身份，写入后不可修改——这决定了 pool 未知的 swap **不能先写后补**，只能整条暂缓。

```
热路径:  pool 缓存 miss
           ├─ Transaction 照常落库 (其 tag 全部来自日志本身)
           └─ SwapRecord 暂缓,原始数据入队

worker:  按池子分组批量消费
           → 一次 eth_call 解析该池子 (token0/token1/fee)
           → 更新 L1/L2/L3
           → 该池子积压的全部 swap 一次性组装 SwapRecord 落库
```

设计要点：

- **面向接口**（`pendingqueue.Queue`），当前为内存实现，可替换为 Kafka 而调用方不变。内存版重启丢积压是可接受的：Transaction 已独立落库，丢失的只是 `swaps` 表的补充行。
- **按池子分组消费**是 RPC 去重的主力机制：同一新池子涌入的 N 笔 swap 只触发一次解析。
- **队列有界，入队永不阻塞热路径**：队满直接放弃暂缓，退化为只写 Transaction。
- **重试有上限**：解析持续失败的池子最终整组丢弃并告警，不会永久占用队列。

---

## 5. RPC 保护层

所有主动发起的 eth_call 都经过统一的保护：

- **共享限流器**：热路径的 token 解析与 worker 的池子解析共用同一份 RPS 预算——分开限流在回溯场景下会叠加超出节点容忍度。
- **singleflight**：并发请求同一对象合并为一次解析。当前架构（单线程扫描 + 单 goroutine worker）下为防御性设计，为未来并发化预留。
- **Multicall3**：一个对象的多个字段查询打包为单次 RPC 往返。

---

## 6. 核心设计决策

**写入时不归一化。** 所有金额一律存 raw 原始值，decimals 换算是读取方的职责（join `dex_tokens`）。由此，SwapRecord 上的 decimals 字段仅是读时优化：能低成本（L1/L2）获取就填，获取不到留空，不为它引入任何 RPC。

**缓存是加速层，不是依赖。** 系统在 Redis 完全不可用时仍能正确工作（L1 + L3），在 L1 冷启动时仍能正确工作（L2/L3 + 兜底解析）。判断任何"缓存操作失败该怎么办"的问题，答案统一是：记日志，继续。

**成本决定同步/异步。** 低频事件（建池）允许同步解析换取数据即时完整；高频事件（swap）的任何缺口都走异步补全，用短暂的数据延迟换取热路径的确定性延迟。

---

## 7. 当前边界

- 该缓存体系当前仅 Ethereum/Uniswap extractor 接入，BSC/Solana/Sui 尚未使用。
- pending queue 为内存实现，跨进程/持久化场景需实现 Kafka 版本。
- 具体参数（TTL、容量、限流值、超时）见各组件代码及 `configs/`，不在本文档维护。
